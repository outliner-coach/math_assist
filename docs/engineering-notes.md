# 구현 경험

## iPad에서 풀이장 대신 주변 글자가 선택됨

- 증상: iPadOS Safari·Chrome에서 펜이나 손가락 입력이 간헐적으로 시작되지 않고 캔버스 아래 `단계 힌트` 글자가 선택된다.
- 원인: 캔버스만 선택 불가로 만들어도 WebKit은 같은 상호작용 영역의 선택 가능한 형제 글자를 선택 대상으로 삼는다. React 포인터 이벤트의 기본 동작 취소만으로 네이티브 선택 경로가 항상 막히지도 않는다.
- 대응: 연습 화면 전체의 `practice-interaction-surface`와 자손에 WebKit 선택·콜아웃 차단을 유지하고 `input`, `textarea`만 선택을 다시 허용한다. 캔버스의 pointer·touch 시작과 이동은 캡처 단계의 non-passive 네이티브 리스너에서 취소한다. 한 `pointerId`만 처리하고 시작점을 즉시 그리며 빠른 입력은 coalesced event를 순서대로 그린다. `setPointerCapture()` 실패는 스트로크 실패로 취급하지 않는다.
- 확인: 컴포넌트 테스트 뒤 WebKit에서 힌트의 `webkitUserSelect`, 빈 `window.getSelection()`, 취소된 `touchstart`, 실제 캔버스 픽셀을 확인하고 마지막에는 실제 iPad에서 검증한다.

## 개발 서버와 빌드가 서로 방해함

- 증상: 개발 서버가 켜진 채 `next build`를 실행하거나 반대로 실행하면 `.next` 관련 오류, 사라진 산출물, 일관되지 않은 화면이 나타난다.
- 원인: 두 프로세스가 같은 `.next` 캐시와 산출물을 동시에 갱신한다.
- 대응: 개발 서버를 종료한 뒤 빌드한다. 브라우저 확인이 계속 필요하면 완성된 `out/`을 별도 HTTP 서버로 제공한다.
- 확인: 남은 Next 프로세스가 없는 상태에서 빌드를 다시 실행하고 정적 경로 수와 종료 코드를 확인한다.

## GitHub Pages 경로에서는 루트 기준 자산이 깨짐

- 증상: 로컬 루트에서는 JSON과 이미지가 보이지만 실제 `/math_assist` 배포에서는 404가 난다.
- 원인: 배포본은 도메인 루트가 아니라 `basePath` 아래에 있으며 정적 데이터와 자산 URL이 접두 경로를 빠뜨렸다.
- 대응: 데이터와 학년 자산은 `NEXT_PUBLIC_BASE_PATH`를 통해 URL을 만든다. 새 코드에 도메인 루트 `/data`나 `/assets`를 직접 넣지 않는다.
- 확인: 로컬 E2E의 `/math_assist` 기준 경로와 배포된 새 브라우저 세션에서 모두 자산 응답과 렌더 화면을 확인한다.

## 정적 HTML 응답만으로 화면 성공을 판단할 수 없음

- 증상: 정적 경로의 HTTP 응답은 성공하지만 저장 기록에 따른 이어하기, 클라이언트 오류, 실제 버튼 상태가 확인되지 않는다.
- 원인: 홈·연습·결과의 주요 상태는 브라우저 hydration과 localStorage를 읽은 뒤 결정된다.
- 대응: 경로 존재는 정적 산출물로 확인하고, 상태 기반 동작은 Playwright나 실제 브라우저에서 확인한다.
- 확인: 새 브라우저와 기록을 주입한 브라우저를 나누어 주요 행동, 콘솔 오류, 저장 변경을 검사한다.

## 이전 진도 형식을 읽을 때 최신 형식으로 덮어쓰면 기록이 손상됨

- 증상: 홈만 방문했는데 예전 완료·복습·선택 단원 기록이 줄거나 다른 형식으로 바뀐다.
- 원인: 홈은 1·2·3·5학년의 서로 다른 저장 형식을 요약할 뿐 소유하지 않는데, 읽기 과정에서 정규화 결과를 다시 쓰면 학년별 마이그레이션 책임이 섞인다.
- 대응: `guest-home`은 기존 이름을 읽기만 하고 `mathAssist_guestHome_v1`에는 선택 학년만 저장한다. 학년별 형식 변경은 해당 진행 모듈의 로더·저장기와 이전 형식 회귀 테스트에서 수행한다.
- 확인: 각 과거 저장 예제를 넣고 홈을 연 뒤 기존 키의 원문이 변하지 않았는지와 이어하기 대상이 맞는지 확인한다.

## 템플릿 수정이 열려 있는 연습에 반영되지 않음

- 증상: JSON 템플릿을 고쳤는데 이미 시작한 문제는 이전 문장·정답·그림을 계속 사용한다.
- 원인: 세션을 만들 때 생성된 문제 전체를 localStorage에 스냅샷으로 저장한다.
- 대응: 검증할 때 기존 활성 세션을 명시적으로 지우거나 새 세트·개념으로 시작한다. 제품에서는 진행 중 세션의 정답을 조용히 바꾸지 않는다.
- 확인: 새 세션의 `templateId`, 문장, 정답, 시각 데이터가 변경본에서 생성됐는지 확인한다.

## 정답 선노출은 보이는 글자만 가려서는 남음

- 증상: 문제 화면에서 결과 숫자를 가렸지만 다른 시각 유형이나 접근성 이름에 정답이 남는다.
- 원인: `visualConfig`의 `result`, `target`, `product` 같은 답 전용 필드를 렌더러가 제각각 해석한다.
- 대응: 문제 카드의 공개 상태를 모든 시각 렌더러에 전달하고, 공개 전에는 답 전용 값이 DOM에 생성되지 않게 한다.
- 확인: 렌더러 집중 테스트와 브라우저 E2E에서 제출 전 부재와 제출 후 존재를 한 쌍으로 검사한다.

## E2E 동시 실행에서 일회성 시간 초과가 발생함

- 증상: 2026-07-21 전체 17개 브라우저 테스트의 첫 실행에서 한 테스트가 숫자 키패드를 기다리다 30초에 종료됐지만, 같은 테스트 단독 실행과 전체 재실행은 코드 변경 없이 통과했다.
- 원인: 재현되지 않아 확정하지 못했다. 여러 작업자가 개발 서버를 시작하는 전체 실행의 초기 로딩 경쟁이 후보지만 증거는 없다.
- 대응: 같은 실패가 다시 나면 실패 영상·추적에서 실제 문제 유형과 페이지 상태를 먼저 확인하고, 무조건 재시도 횟수만 늘리지 않는다.
- 확인: 실패 테스트 단독 실행 뒤 전체 17개를 같은 설정으로 다시 실행해 단독·전체 결과를 함께 기록한다.

## 세 도형 겹침 그림의 고정 SVG가 문제 수치와 모순됨

- 증상: `triple-overlap-inclusion` 예시에서 계산상 `BC만` 영역은 0인데 고정 다각형 SVG에는 B와 C만 겹친 영역이 보였다. 단독 영역의 시각 비율도 문제 수치와 크게 달랐다.
- 원인: 문제 문장과 정답은 `shapeArea`, `exclusiveAreas`, `tripleOverlap`에서 계산했지만 그림은 이 값과 무관한 세 다각형 좌표를 사용했다. 고정 좌표를 조금 조정하는 접근은 다른 매개변수에서 같은 오류를 반복하며, 이중 겹침값을 직접 라벨링하면 중간 답을 선노출한다.
- 대응: 세 도형의 공통 넓이 식을 풀어 `A만`, `B만`, `C만`, `AB만`, `AC만`, `BC만`, `ABC`의 일곱 영역 모델을 만든다. 새 문제는 이 모델을 저장하고, 구형 세션은 렌더 시 같은 함수로 모델을 복원한다. SVG는 한 칸 1단위 넓이의 같은 크기 셀을 그려 비율을 정확히 보존하고, 0인 영역은 DOM에 만들지 않는다. 문제에서 주지 않은 이중 겹침값은 숫자 라벨로 표시하지 않는다.
- 확인: 모델 단위 테스트에서 각 도형 넓이 재구성, 음수·비정수 거부, 합집합 넓이를 검사한다. 렌더러는 영역별 셀 수, 0 영역 부재, 중간 답 라벨 부재를 검사한다. 템플릿 검증기는 모든 파라미터 표본의 가능성을 검사하며, E2E는 새 payload와 `model`·`semantics`가 없는 구형 저장 세션을 모두 확인한다. 모바일 390×844와 태블릿 1024×768에서 가로 넘침과 콘솔 오류가 없어야 한다.

## 문제 템플릿 산술을 허용 문자 검사 뒤 `eval`로 실행함

- 증상: 숫자와 산술 기호만 남기는 정규식이 있었지만 문제 생성기가 템플릿 식을 `eval`로 실행해 엄격한 CSP와 정적 코드 검토에서 차단 요소가 됐다.
- 원인: 템플릿의 등록 함수 호출을 먼저 값으로 치환한 뒤 남은 산술식의 우선순위와 단항 부호를 처리할 작은 평가기가 없었다. 품질 감사의 임시 TypeScript 로더도 생성기의 직접 의존 파일을 수동으로 복사해 새 모듈 추가 시 검증 명령이 깨질 수 있었다.
- 대응: 유한 십진수, 괄호, `+ - * /`, 단항 부호만 해석하는 길이·중첩 제한 재귀 하강 parser를 별도 모듈로 둔다. 지수, 식별자, `NaN`, `Infinity`, 0 나눗셈, 후행 토큰, 비유한 결과는 거부한다. 생성기는 기존 등록 함수 처리 뒤 이 parser만 호출하고, 감사 로더는 parser 모듈도 같은 임시 디렉터리에 변환한다.
- 확인: parser 허용·거부 쌍 테스트, 등록 함수 인자 산술, 거부 식의 미평가 표식, 모든 5학년 템플릿의 A/B/C 10문제 결정론 생성, 6학년 5/10문제 생성, `validate:templates`, `validate:grade6`, `audit:problems`을 함께 실행한다.

## 연습 초기화가 같은 요청의 무작위 세션을 두 번 생성함

- 증상: 세 도형 겹침 E2E를 병렬 반복하면 localStorage에서 찾은 겹침 문제 인덱스를 눌렀는데 화면에는 같은 인덱스의 다른 넓이 문제가 나타났다. 수정 전 재현은 10회 중 2회 실패했으며, 더 단순한 단독 실행은 통과해 문제를 숨겼다.
- 원인: React Strict Mode의 개발용 mount-cleanup-remount가 비동기 초기화 effect를 두 번 시작했다. cleanup 뒤에도 첫 실행이 `await`에서 돌아와 별도의 무작위 문제 묶음을 저장했고, 브라우저 검사가 첫 저장을 읽은 직후 두 번째 실행이 저장과 화면 상태를 교체할 수 있었다. 병렬 개발 서버 부하는 이 창을 넓혔지만 원인은 서버 간 저장 공유가 아니라 한 탭 안의 stale effect였다.
- 대응: effect 실행마다 활성 플래그를 두고 release gate, 개념, 템플릿의 각 비동기 경계 뒤에 현재 실행인지 확인한다. cleanup된 실행은 localStorage, 화면 세션, 차단 상태, 오류, 로딩 상태를 변경하지 않는다. 기존 `saveSession`을 그대로 사용하므로 5·6학년 분리와 손상 저장소 덮어쓰기 거부 계약은 유지된다.
- 확인: Strict Mode 컴포넌트 테스트에서 개념 조회는 두 번 시작되더라도 템플릿 생성·세션 저장·화면 반영은 활성 실행 한 번만 수행되는지 확인한다. 실제 겹침 E2E는 재시도 없이 6 workers로 10회 이상 반복하며 저장 스냅샷으로 고른 문제가 화면에도 나타나는지 확인한다.
- 유지 계약: 새 `await`를 초기화 effect에 추가할 때는 그 결과로 상태나 저장소를 바꾸기 전에 활성 실행 여부를 확인한다. 취소된 실행의 `finally`도 로딩 상태를 끝내면 안 되며, 불안정성을 재시도 횟수 증가나 테스트 대기로 가리지 않는다.
