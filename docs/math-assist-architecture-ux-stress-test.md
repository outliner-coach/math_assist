# Math Assist 아키텍처·아동 UX 스트레스 테스트

작성일: 2026-07-20
대상: 현재 저장소와 `math-assist-production-proposal.html`에 제안된 1–6학년 확장안

## 결론

제안의 방향은 맞지만 현재 범위는 한 릴리스에 담기에는 과합니다. 공통 학습 루프, 전 학년 풀이장, 가벼운 미션 프레이밍, 캐릭터 안내는 서로 잘 맞습니다. 그러나 계정·다중 기기 동기화·고급 풀이 도구·캐릭터 꾸미기·여러 보상 지표·4·6학년 콘텐츠까지 동시에 묶으면 아이가 체감하는 개선보다 상태와 마이그레이션 복잡도가 먼저 커집니다.

권장 방향은 다음 한 문장으로 요약됩니다.

> **채점 가능한 학습 활동을 코어로 두고, 미션·캐릭터·보상은 코어를 가리지 않는 파생 경험으로 둔다.**

## 1. 현재 코드에서 확인한 구조

현재 Math Assist는 하나의 공통 제품 위에 학년군별 표현만 다른 상태가 아닙니다. 실질적으로 두 종류의 학습 앱과 네 종류의 진도 저장 방식이 함께 있습니다.

| 영역 | 현재 구현 | 구조적 의미 |
|---|---|---|
| 1학년 | 단일 탐험 화면, 스테이지·보상·힌트·복습 | 미션 앱 |
| 2학년 | 단원 선택 + 단원별 미션, 숙련 별·보상 | 미션 앱의 별도 구현 |
| 3학년 | 단원 선택 + 미션, 구조화 답안·보상 | 2학년과 유사하지만 별도 상태·컴포넌트 |
| 5학년 | 개념 설명 + 고정 10문제 세션 + 결과 | 연습 세션 앱 |
| 진도 | `mathAssist_grade1Progress`, `grade2Progress`, `grade3Progress`, `progress_v1` | 홈이 학년별 어댑터로 네 형식을 직접 해석 |
| 풀이장 | 5학년 연습 화면에만 렌더링, 문제 이동 시 초기화 | 공통 학습 상태가 아니라 일회성 UI |
| 세션 | `mathAssist_currentSession` 하나, 2시간 만료 | 한 시점에 한 연습 세션만 복구 가능 |

근거가 되는 주요 코드는 다음과 같습니다.

- `src/lib/guest-home.ts`: 학년별 저장 키와 서로 다른 진도 모양을 각각 해석합니다.
- `src/lib/grade1-progress.ts`, `grade2-progress.ts`, `grade3-progress.ts`: 유사한 저장·정규화·복구 로직이 학년별로 복제되어 있습니다.
- `src/lib/session.ts`: 5학년 연습은 전역 세션 키 하나를 사용합니다.
- `src/app/grade/1/Grade1GameClient.tsx`, `grade/2/Grade2GameClient.tsx`, `grade/3/Grade3GameClient.tsx`: 문제 선택, 답안, 힌트, 보상, 저장 상태가 큰 화면 컴포넌트 안에 결합되어 있습니다.
- `src/app/practice/[conceptId]/PracticeClient.tsx`: 5학년은 별도의 10문제 세션 계약을 사용합니다.
- `src/components/ScratchPad.tsx`: 펜·지우개·전체 지우기는 있지만 획 저장, 되돌리기, 문제 복구 계약은 아직 없습니다.

따라서 전 학년 기능을 바로 추가하기보다 먼저 공통 활동·진도 경계를 만드는 것이 맞습니다.

## 2. 아이가 실제로 경험해야 할 한 개의 루프

모든 기능은 다음 다섯 단계 중 하나를 더 쉽게 만들어야 합니다.

1. **바로 시작** — 홈에서 한 개의 추천 활동을 누릅니다.
2. **생각하고 표현** — 문제를 읽고 보기·숫자·풀이장으로 생각을 표현합니다.
3. **즉시 확인** — 정오와 짧은 이유를 확인합니다.
4. **안전하게 재도전** — 틀려도 손실 없이 힌트나 다른 표현으로 다시 풉니다.
5. **작게 성취하고 계속** — 미션 진행이 한 칸 움직이고 다음 문제를 선택합니다.

캐릭터, 미션, 보상, 숙달은 이 루프의 결과를 설명하는 표현이어야 합니다. 이들 중 어느 것도 문제 로드, 답안 저장, 채점, 다음 문제 진입을 막아서는 안 됩니다.

### 제품 수준의 마찰 예산

- 학년 선택 또는 이어하기에서 문제 도달까지 **주요 탭 3회 이내**를 목표로 합니다.
- 문제 화면에는 언제나 **주요 행동 1개**만 강조합니다: 정답 확인 또는 다음 문제.
- 로그인, 캐릭터 선택, 안내 투어, 보상 수령을 학습 시작의 필수 단계로 두지 않습니다.
- 문제·답 입력·풀이장은 태블릿에서 같은 맥락에 남아야 하며, 도구 패널이 문제를 밀어내지 않아야 합니다.
- 틀린 뒤에는 감점·연속 기록 손실·보상 회수 대신 **한 문장의 회복 안내와 재도전**을 제공합니다.
- 보상 애니메이션은 문제마다 흐름을 끊지 않고 짧은 미션 종료 시점에 묶습니다.

## 3. 동기와 재미를 만드는 요소

재미는 화면에 게임 요소가 많은 상태가 아니라 아이가 수학 활동 안에서 네 가지 감각을 얻는 상태입니다.

| 욕구 | 제품 장치 | 피해야 할 대체물 |
|---|---|---|
| 유능감 | 풀 수 있는 난이도, 즉시 피드백, 보이는 진전 | 점수·XP 숫자만 증가 |
| 선택감 | 추천 1개 + 선택 가능한 다른 활동 2–3개 | 끝없는 지도, 잠긴 필수 콘텐츠 |
| 의미 | 짧은 미션 문맥, 문제와 연결된 시각 모델 | 수학과 무관한 긴 서사 |
| 관계감 | 한 캐릭터의 상황별 격려와 전략 안내 | 여러 캐릭터의 동시 대화·팝업 |

### 학년군별 적정 강도

| 학년군 | 미션 길이 | 캐릭터 | 풀이장 | 보상 노출 |
|---|---:|---|---|---|
| 1–2 | 1–3문제 | 큰 전신, 짧은 음성/문장 없이도 이해되는 제스처 | 펜·지우개·되돌리기, 수직선 또는 수블록 중 문제 맞춤 1개 | 미션 종료 스티커 1개 |
| 3–4 | 3–5문제 | 작은 동반자, 요청할 때 전략 힌트 | 펜·지우개·되돌리기 + 문제 맞춤 도구 1개 | 챕터 진행과 배지 1종 |
| 5–6 | 5문제 기본, 10문제 선택 | 절제된 코치, 헤더·피드백에만 등장 | 펜·지우개·되돌리기, 격자/자 등 필요 도구 | 숙달 진전 중심, 꾸미기 보조 |

고학년의 10문제 세트는 없애기보다 선택 가능한 집중 모드로 두는 편이 좋습니다. 저학년과 처음 복습하는 아이에게는 짧은 완료 경험이 더 중요합니다.

## 4. 과도한 부분과 범위 결정

### 지금 유지

- 결정적 문제 생성·채점·정오 판단
- 로그인 없는 학년 선택과 한 개의 이어하기 행동
- 문제별 즉시 피드백, 힌트, 손실 없는 재도전
- 전 학년 공통 풀이장 V1: 펜·지우개·되돌리기·지우기·문제별 임시 저장
- 짧은 미션 시작·완료 셸
- 세 동물 중 한 아바타 선택, 화면 맥락에는 한 캐릭터만 노출
- 완료 문제·복습 필요·최근 활동을 보여 주는 공통 진도 뷰

### 단순화해서 유지

| 제안 기능 | 단순화 |
|---|---|
| 캐릭터 표정·제스처 세트 | 디자인 자산은 9상태로 만들되 런타임 V1은 `환영·생각·힌트·회복·축하` 5상태만 연결 |
| 아바타·꾸미기 | 세 캐릭터 선택만 먼저 제공. 의상 상점·복수 화폐·희귀도는 보류 |
| 미션 지도 | 탐험 가능한 월드 엔진 대신 챕터 카드와 한 줄 경로 사용 |
| 보상 | XP·레벨·연속일·일일 목표·숙련 별·수집품을 동시에 노출하지 않음. 아이 화면은 `경로 진전 + 미션 보상 1종`으로 제한 |
| 숙달 | 새 점수 알고리즘보다 현재 정오·첫 시도·복습 결과를 네 단계 설명으로 투영 |
| 풀이 도구 | 모든 조작물을 한꺼번에 넣지 않고, 기본 3도구 + 문제 유형별 보조 도구 1개 |
| 보호자 기능 | 초기에는 기록 연결·복구만. 상세 대시보드와 주간 리포트는 사용 데이터 뒤에 결정 |

### 후속 근거가 생길 때까지 보류

- 리더보드, 친구 경쟁, 채팅, 선물
- 연속 학습 실패에 대한 손실·강등
- 여러 게임 화폐, 아이템 능력치, 확률형 보상
- 탐험 가능한 월드, 사이드 퀘스트, 보스 시스템
- AI 필기 인식·자동 풀이 판정
- 풀이장 다중 페이지·무제한 장기 보관
- 실시간 운영 CMS와 복잡한 실험 플랫폼
- 모든 풀이 획의 다중 기기 동기화. 먼저 완료·진도·선택 아바타를 동기화하고 실제 필요를 측정

## 5. 권장 아키텍처

### A. Deterministic Learning Core

콘텐츠, 문제 생성, 답 정규화, 채점, 해설은 순수 도메인 로직으로 유지합니다. 미션이나 캐릭터 상태가 정답 판단에 들어오지 않습니다.

### B. Learning Activity Contract

1문제 미션과 10문제 세트를 같은 활동 계약으로 표현합니다.

```ts
interface LearningActivitySession {
  sessionId: string
  learnerId: string
  grade: number
  activityId: string
  mode: 'mission' | 'practice' | 'review'
  items: ActivityItemSnapshot[]
  responses: ActivityResponse[]
  currentIndex: number
  status: 'active' | 'completed' | 'expired'
  startedAt: number
  updatedAt: number
  expiresAt: number
}
```

학년별 화면은 이 계약을 소비하되 문제 수, 텍스트 밀도, 캐릭터 크기만 프리셋으로 바꿉니다.

### C. Attempt Receipt + Progress Projection

대규모 이벤트 소싱 대신 중복 보상과 동기화 병합에 필요한 작은 시도 영수증만 남깁니다.

```ts
interface AttemptReceipt {
  attemptId: string
  learnerId: string
  activityId: string
  itemId: string
  variantKey: string
  correct: boolean
  usedHint: boolean
  completedAt: number
}
```

완료 수, 복습 필요, 숙달, 미션 경로, 보상은 영수증과 콘텐츠 정의에서 계산한 **projection**으로 둡니다. 캐릭터의 현재 표정과 화면 애니메이션은 저장하거나 동기화하지 않습니다.

### D. Experience Preset

학년별 컴포넌트 안에 조건문을 흩뿌리지 않고 세 가지 프리셋을 사용합니다.

```ts
interface ExperiencePreset {
  ageBand: 'play' | 'bridge' | 'study'
  defaultMissionLength: number
  mascotMode: 'full' | 'companion' | 'coach'
  rewardCadence: 'mission' | 'chapter' | 'mastery'
  scratchTools: ScratchToolId[]
  textDensity: 'low' | 'medium' | 'high'
}
```

프리셋은 표현을 바꾸지만 채점과 진도 의미는 바꾸지 않습니다.

### E. Persistence Ports

UI가 localStorage 키를 직접 알지 않도록 아래 저장소 경계를 둡니다.

- `LearningSessionRepository`
- `ProgressRepository`
- `SketchRepository`
- `LearnerProfileRepository`

먼저 localStorage 구현과 기존 키 읽기 어댑터를 만듭니다. 서버 동기화는 같은 인터페이스의 두 번째 구현으로 붙입니다. 이렇게 해야 로그인 일정이 늦어져도 공통 학습 경험은 진행할 수 있습니다.

### 동기화 우선순위

1. 완료 시도 영수증, 복습 상태, 최근 활동
2. 선택 아바타와 획득한 단일 보상 ID
3. 활성 세션 복구
4. 필요가 확인된 문제의 압축 획 데이터

캐릭터 표정, 팝업 열림, 지도 스크롤 위치, 애니메이션 상태는 동기화하지 않습니다.

## 6. 권장 출시 순서

### Release 0 — 핵심 루프와 저장 경계

- 공통 `LearningActivitySession`과 학년별 어댑터
- 기존 네 진도 키를 읽는 `ProgressRepository`
- 공통 홈의 한 개 이어하기 행동
- 1·2·3·5학년 회귀 테스트

성공 조건: 기존 기록 손실 없이 네 학년 모두 같은 `resume / completed / review` 뷰를 생성합니다.

### Release 1 — 재미를 검증하는 얇은 공통 셸

- 전 학년 Scratchpad V1
- 짧은 미션 시작·완료 셸
- 세 캐릭터 선택과 5개 반응 상태
- 경로 진전 + 보상 1종

성공 조건: 아이 관찰 테스트에서 문제 도달 3탭 이내, 주요 행동 혼동 없음, 재도전률 저하 없음.

### Release 2 — 콘텐츠와 기록 연결

- 3학년 안정화, 4학년 핵심 단원
- 보호자 계정과 게스트 기록 연결
- 완료·복습·아바타 우선 동기화

성공 조건: 로그인 없이도 학습 가능하고, 기록 연결 실패 시 로컬 백업에서 복구됩니다.

### Release 3 — 근거가 있는 확장

- 5문제/10문제 선택형 연습, 6학년 콘텐츠
- 실제 사용된 풀이 도구만 확장
- 캐릭터 꾸미기와 풀이 획 동기화는 관찰 데이터로 우선순위 결정

성공 조건: 기능 수가 아니라 주간 자발적 재방문, 미션 완료, 오답 재도전, 저장 복구 지표가 개선됩니다.

## 7. 구현 전 사용자 검증

각 학년군 5명 정도의 소규모 관찰만으로도 큰 마찰을 조기에 찾을 수 있습니다.

1. 아무 설명 없이 학년 선택 후 문제를 시작하게 합니다.
2. 한 문제를 풀고, 틀렸을 때 다시 시도하게 합니다.
3. 풀이장을 열고 답을 바꾸게 합니다.
4. 앱을 닫았다가 이어서 풀게 합니다.
5. 보상을 받은 뒤 다음에 무엇을 할지 관찰합니다.

관찰자는 탭 수, 멈춘 위치, 잘못 누른 행동, 도움 요청, 재도전 여부를 기록합니다. “재미있었어?”라는 질문보다 아이가 스스로 다음 문제를 누르는지가 더 강한 신호입니다.

## 최종 판단

Math Assist의 차별점은 거대한 게임 세계가 아니라 **아이의 손으로 생각할 수 있는 수학 문제 + 안전한 재도전 + 작게 이어지는 원정**이 되어야 합니다. 아키텍처도 이 순서를 따라야 합니다. 학습 코어와 저장 경계를 먼저 얇게 통일하고, 동기 장치는 파생 계층으로 붙이며, 실제 사용 근거가 없는 세계·화폐·동기화 범위는 보류하는 것이 프로덕션 수준에 더 가깝습니다.
