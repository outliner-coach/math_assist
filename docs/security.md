# 학습 기록 보호 정책

## 보호 대상과 현재 경계

실제 배포본은 정적 사이트이며 서버 계정과 원격 저장이 없다. 2026-07-21 커밋 `14a0904`와 Pages 실행 `29792892987`로 새 시도 receipt와 문제별 풀이장 획까지 기기 로컬 보호 대상으로 배포했다. receipt는 원답 문자열과 풀이장 획을 포함하지 않고, 풀이장 문서는 정답·답안을 포함하지 않는다. 답안, 활성 세션, 결과, 완료·복습·숙련 기록, 최근 활동, 선택 학년을 포함한 이 정보는 현재 기기의 같은 브라우저에서만 읽고 쓰며 공개 앱이 원격 서버나 분석 서비스로 보내지 않는다.

브라우저 사용자는 현재 브라우저 프로필의 같은 origin에 저장된 기록을 읽고 갱신하고 초기화할 수 있다. 다른 기기나 다른 브라우저 프로필의 기록에는 접근할 수 없지만, 같은 브라우저 프로필을 여러 어린이가 공유하면 하나의 고정 키 집합을 함께 보게 된다. 현재는 학습자별 분리가 없다. 브라우저 데이터를 지우면 기록이 사라지며 현재 배포본에는 복구 경로가 없다.

## 현재 접근 흐름

| 사용자 | 행동 | 허용 조건 | 명시적으로 허용하지 않는 것 |
|---|---|---|---|
| 방문자 | 공개 학습 콘텐츠 열기 | 정적 경로가 존재함 | 존재하지 않는 학년·개념을 학습 가능 상태로 가장하기 |
| 게스트 학습자 | 답안·진도·세션 저장 | 브라우저 localStorage 사용 가능 | 다른 기기 또는 다른 브라우저 프로필의 기록 읽기; 같은 프로필 안에서 학습자별 분리는 제공하지 않음 |
| 게스트 학습자 | 기록 초기화 | 해당 학년 화면에서 명시적으로 확인 | 다른 학년의 저장 영역을 함께 삭제하기 |
| 운영자 | 정적 콘텐츠 배포 | 검증된 커밋이 `main`에 반영됨 | 배포본에서 개별 학습자 기록 열람하기 |

localStorage가 차단되거나 손상되면 앱은 기본 상태로 복구하거나 저장 불가를 알려야 한다. 손상된 원문은 명시적 초기화 전까지 보존하고 자동 저장으로 덮어쓰지 않는다. 손상된 한 학년의 기록 때문에 다른 학년 기록까지 지우면 안 된다. 현재 인증 성공·실패나 학습 행동을 수집하는 서버 감사 기록은 없으며, 정적 배포 로그를 학습자 활동 기록으로 사용하지 않는다.

4학년 `mathAssist_grade4Progress`를 읽을 수 없거나 schema가 맞지 않으면 원문을 즉시 삭제하지 않는다. 그 storage 객체에서는 자동 저장도 중단하여 빈 기본 상태가 손상 원문을 덮어쓰지 못하게 하고, 학습자가 4학년 화면에서 초기화를 두 번 확인한 경우에만 4학년 키를 새 schema로 교체한다. 이 과정은 1·2·3·5학년 키를 읽기만 하며 변경하지 않는다.

5·6학년 progress/session/result가 손상되거나 학년·문항 수 계약과 맞지 않으면 읽기 결과만 빈 상태로 격리한다. 자동 학습 저장, 새 세션 생성, 완료 후 정리 경로는 그 원문을 덮어쓰거나 삭제하지 않는다. Grade 6 화면은 손상 session/result를 설명하고 해당 Grade 6 키만 초기화하는 명시적 복구 행동을 제공한다. 저장 계층의 명시적 reset/clear 뒤에만 새 값을 쓸 수 있으며, 결과와 진도 저장이 모두 성공하기 전에는 활성 세션을 삭제하지 않는다. 정상인 Grade 5 이전 세션의 누락된 `grade`와 `itemCount`는 각각 5와 10으로 읽고 원문을 재작성하지 않는다.

`mathAssist_attemptReceipts_v1`은 기존 학년별 진도와 분리된 추가 원장이다. 이 원장이 손상되거나 쓰기에 실패해도 앱은 기존 진도 키를 초기화하거나 축소하지 않으며, 손상된 원장을 자동으로 빈 원장으로 덮어쓰지 않는다.

풀이장 저장은 `learnerId + sessionId + itemId`로 학년·활동·문항을 격리한다. 저학년의 명시적 재시작 순번은 각 학년 progress key의 선택적 `missionSketchRunOrdinal` 필드로만 보존하며 화면이 별도 localStorage 이름을 만들지 않는다. 누락·잘못된 순번은 0으로 정규화하되 같은 progress의 완료·복습·보상 필드는 유지한다. 손상된 한 문서는 그 문제의 빈 문서로만 복구하고 다른 풀이장과 학년별 진도를 삭제하지 않는다. 이 빈 복구 화면에서 새로 그려도 손상 원문은 자동으로 덮어쓰지 않고 저장 불가를 알린다. 완료·만료 활동의 문서는 읽기 전용이며, 풀이장 획은 현재 원격 동기화·backup 범위에 포함하지 않는다.

현재 작업트리의 6학년 비와 비율 단원은 공개 원장의 `releaseState.grade6: released`로 열려 있다. 그러나 원장 상태가 정확히 `released`가 아니면 `/grade/6`, Grade 6 unit·concept·practice·result·retry는 저장 repository 호출 전에 같은 준비 중 화면으로 fail-closed한다. 원장 부재, schema 오류, fetch 실패, 알 수 없는 상태도 모두 닫으며 기존 Grade 5/6 저장 원문을 읽어 노출하거나 변경하지 않는다.

## 기록 저장 계정의 목표 접근 흐름

1. 게스트가 먼저 학습한다. 계정 화면을 통과해야 문제를 볼 수 있게 만들지 않는다.
2. 학습자는 `기록 저장하기`를 직접 선택할 수 있다. 원격 기록을 생성하기 전에는 적용 가능한 적법 근거와 법정대리인 동의·동의자 확인을 완료하고, 그 뒤 서버가 추측하기 어려운 학습 번호를 생성하며 학습자는 4자리 숫자 비밀번호를 정한다.
3. 계정 생성이 성공하기 전에는 로컬 기록을 삭제하거나 원격 저장 완료로 표시하지 않는다. 성공 후에도 첫 동기화 결과를 확인할 때까지 로컬 사본을 유지한다.
4. 처음 만든 기기는 취소 가능한 로그인 세션을 유지한다. 다른 기기는 학습 번호와 비밀번호가 모두 맞아야 접근한다.
5. 잘못된 학습 번호와 잘못된 비밀번호는 같은 실패 문구를 보여 준다. 반복 실패는 계정과 접속 지점을 기준으로 점차 지연하고 일시 차단한다.
6. 만료·철회·로그아웃된 세션은 다시 비밀번호 확인을 거쳐야 한다. 세션 철회가 실패한 상태에서 로그아웃 완료로 표시하면 안 된다.

## 비공개 staging 동의 provisioning 경계

원격 learner/account 생성은 일반 auth core를 직접 호출하지 않고 서버 전용
consent provisioning service를 통과한다. service는 법정대리인 동의와 동의자
확인이 완료됐음을 나타내는 one-time authorization artifact를 다음 조건으로
검증한다.

- artifact는 `authorized`, `consumed`, `revoked` 상태와 발급·동의·확인·만료
  시각, 원격 학습 기록 목적, 승인된 정책 버전, 확인 receipt hash를 가진다.
- raw authorization, pre-account learner binding secret, guardian 이름·이메일·
  전화번호는 저장하지 않는다. authorization과 learner binding은 SHA-256 hash로,
  감사 subject는 별도 서버 pepper의 HMAC으로만 다룬다.
- authorization은 발급 시 결합된 같은 learner request에서만 사용할 수 있다.
  다른 binding, 미존재·만료·철회·소비된 token, 목적·정책 버전 불일치는 같은
  실패로 거부하고 remote learner/account를 만들지 않는다.
- authorization 소비와 auth account+첫 session 생성은 provider의 같은 원자
  transaction에 포함한다. auth 생성 또는 authorization 최종화가 실패하면 token은
  미소비 상태이고 account/session은 0이어야 한다.
- 이 artifact는 동의의 법적 충분성을 스스로 증명하지 않는다. 실제 guardian
  확인 수단, 적용 가능한 적법 근거, 증적 접근·보존·철회·삭제 정책이 승인되기
  전에는 production 기능 플래그나 공개 UI/API를 열지 않는다.

## 비공개 원격 기반의 현재 상태와 배포 차단 조건

- remote merge, rollback backup, 엄격한 envelope 검증, mock transport, auth core, one-time recovery, consent provisioning은 서버·저장소 경계를 검증하기 위한 비공개 기반이다. production feature flag는 `false`이며 공개 route·UI·실제 provider는 없다.
- 현재 in-memory repository와 limiter는 production provider가 아니다. 실제 구현은 동의 소비와 account·첫 session 생성을 같은 transaction으로 묶고, distributed rate limit, 서버가 도출한 network identifier, 절대·유휴 session 만료, TLS, secure cookie/header, CORS/CSRF, audit 보존·복구를 제공해야 한다.
- account 삭제 후 보존 기간, 보호자 연결 해제, session 만료, guardian 확인 수단, 적법 근거, 동의 증적 접근·철회·삭제, 아동용 고지가 승인되어야 한다.
- 현재 의존성 보안 감사에서 보고된 Next.js·KaTeX·Playwright 계열 advisory를 지원 버전으로 갱신하고 `npm audit` 0 또는 문서화된 예외를 승인해야 한다. 정적 배포에서 비활성인 서버 공격면이라도 원격 API 공개 근거로 무시하지 않는다.
- 위 조건 중 하나라도 남아 있으면 feature flag를 켜거나 임시 API를 공개하지 않는다. 기반 테스트 통과는 production 인증·개인정보 처리 승인이 아니다.

## 자격 정보의 수명

- 4자리 비밀번호 원문을 브라우저 영구 저장소, 서버 데이터, 로그, 오류 보고서에 저장하면 안 된다. 서버는 느린 비밀번호 검증 방식과 계정별 임의값으로 만든 검증값만 보관해야 한다.
- 로그인과 복구 정보는 암호화된 연결에서만 전송한다. 공개 클라이언트 설정에 서버 비밀값을 넣지 않는다.
- 로그인 세션은 서버에서 철회할 수 있어야 하며 로그아웃, 비밀번호 재설정, 계정 삭제 시 기존 세션을 모두 무효화한다.
- 복구 코드는 가입 때 한 번만 원문을 보여 주며 서버에는 검증값만 남긴다. 성공적으로 사용한 복구 코드는 즉시 폐기하고 새 복구 카드를 발급한다.
- 비밀번호를 잊었을 때 복구 카드와 연결된 보호자가 모두 없으면 이름·학년·학습 내역을 물어 신원을 추정하지 않는다.

계정 삭제 후 기록 보존 기간과 보호자 연결 해제 후 처리 방식은 아직 결정되지 않았다. 이 두 정책, 실제 세션 만료 시간, 아동 원격 개인정보 처리의 적법 근거·법정대리인 동의 확인·아동용 고지가 승인되기 전에는 원격 로그인 기능을 배포하면 안 된다.

## 최소 수집과 동기화

원격 저장이 시작되어도 실제 이름, 이메일, 휴대폰, 학교, 위치, 광고 식별자를 받지 않는다. 별명은 선택 사항이다. 최초 동기화 대상은 완료 기록, 복습 상태, raw answer가 없는 시도 receipt, 최근 활동, 선택 아바타이며 풀이장 획·원답·캐릭터 표정·팝업 상태·스크롤 위치·애니메이션 상태는 보내지 않는다. 원격 envelope는 런타임에서 허용 필드·배열 크기·식별자 길이·시각·학습자 결합을 검사하며 알 수 없는 필드가 있으면 병합하지 않는다.

기록 병합 전에는 로컬과 원격의 차이를 미리 보여 주고 복구 가능한 기기 로컬 rollback 백업을 만든다. 이 백업의 현재 Grade 5/6 active session 원문에는 기기에서 작성 중인 답이 있을 수 있지만 원격으로 전송하지 않는다. 제출 결과와 사용자 답 스냅샷을 가진 Grade 6 last result, 풀이장, 계정 비밀은 backup allowlist에서 제외한다. 충돌을 조용히 덮어쓰거나 더 오래된 원격 기록으로 최근 로컬 진도를 되돌리면 안 된다.

## 기록해야 할 인증 사건

원격 계정을 도입하면 계정 생성, 로그인 성공과 실패, 일시 차단, 세션 철회, 로그아웃, 비밀번호 재설정, 복구 코드 사용, 보호자 연결·해제, 계정 삭제 요청과 완료를 기록한다. 사건 기록에는 시각, 사건 종류, 익명화된 계정 식별자, 결과만 남기며 비밀번호, 복구 코드, 답안, 풀이장 내용은 남기지 않는다.
