# 큰 코드 이동 대신 작업영역 협업

## 상황

앱은 한 패키지 안에 있고 공용 타입·생성기·저장 파일을 여러 기능이 함께 사용한다. 병렬 작업을 위해 전체 소스를 새 기능 트리로 옮기면 대부분 파일의 경로와 가져오기가 한꺼번에 바뀌어 충돌과 회귀 위험이 커진다.

## 결정

제품 코드는 현재 `app / components / lib / public/data` 구조에 유지한다. 작업은 콘텐츠, 학습 흐름, 화면·시각, 품질 자동화의 네 작업영역으로 나누고 공유 파일을 바꾸기 전에 소유권과 계약을 기록한다. 외부 근거와 작업 인수인계는 코드와 분리된 전용 디렉터리에 남긴다.

## 선택하지 않은 안

- 기능별 전체 재배치는 장기적으로 경계가 더 선명할 수 있지만 현재 규모에서는 대량 이동이 실제 기능 가치 없이 충돌 면적을 키워 선택하지 않았다.
- 경계 없이 모든 작업자가 공용 파일을 직접 수정하는 방식은 같은 타입과 템플릿을 서로 다른 가정으로 바꿀 수 있어 선택하지 않았다.

## 결과

공용 파일 변경에는 조정 비용이 들고 일부 디렉터리는 여러 기능을 계속 포함한다. 대신 기능 작업이 대량 경로 변경과 섞이지 않으며, 공유 계약 변경을 별도로 검토할 수 있다. 구조가 실제로 독립 빌드 단위로 커지면 이 결정을 다시 검토할 수 있다.
