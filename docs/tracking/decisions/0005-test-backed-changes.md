# 테스트를 동반한 제품 변경

## 상황

문제 생성, 정답 노출, 로컬 기록 마이그레이션은 작은 수정도 학생에게 잘못된 문제나 기록 손실을 만들 수 있다. 작업 지침만으로는 빠른 구현 중 테스트가 빠지는 일을 막기 어려웠다.

## 결정

`src/**`와 `public/data/**` 변경은 같은 변경에 집중 테스트를 추가하거나 갱신해야 하며 자동 검사에서 제품 파일만 바뀐 작업을 거부한다. 콘텐츠는 학년별 검증기와 품질 감사, 사용자 흐름은 브라우저 테스트, 전체 제품은 린트·단위 테스트·정적 빌드로 검증한다.

## 선택하지 않은 안

- 작업자가 필요하다고 느낄 때만 테스트하는 방식은 정답 선노출과 이전 진도 손실처럼 재현 가능한 결함을 다시 허용하므로 선택하지 않았다.
- 브라우저 테스트만으로 모든 조합을 확인하는 방식은 수백 개 문제 매개변수와 정규화 경계를 충분히 탐색하기 느려서 선택하지 않았다.

## 결과

기능 변경은 테스트 작성 비용을 동반하고 테스트 의미가 없는 유지보수는 명시적 검토가 필요하다. 반대로 수학 규칙은 빠른 단위·감사 테스트로 넓게 검사하고, 실제 화면 공개 시점과 복구는 소수의 브라우저 흐름으로 확인할 수 있다.
