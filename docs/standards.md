# 변경 규칙

## 의존성과 실행 환경

- 의존성은 `package-lock.json`과 함께 npm으로 관리한다. 의존성 변경 후 잠금 파일이 바뀌지 않거나 `npm ci`가 실패하면 변경을 받을 수 없다.
- GitHub Pages 빌드와 같은 Node.js 24 계열에서 빌드되어야 한다. 로컬에서만 동작하고 배포 빌드에서 정적 내보내기가 실패하면 완료가 아니다.
- 공개 클라이언트 설정에는 `NEXT_PUBLIC_BASE_PATH`처럼 브라우저에 공개되어도 되는 값만 둔다. 비밀번호 검증 비밀값이나 서버 자격 정보가 `NEXT_PUBLIC_` 이름으로 들어가면 위반이다.

## 변경과 검증

- `src/**` 또는 `public/data/**`를 바꾸는 작업은 같은 변경에 해당 동작을 실패시킨 뒤 통과시키는 집중 테스트를 추가하거나 갱신해야 한다. 제품 파일만 바뀌면 `.codex/hooks/tdd-guard.sh`가 거부해야 한다.
- 테스트 의미가 없는 유지보수에서만 검토 후 `TDD_GUARD_ALLOW_NO_TEST=1`을 사용할 수 있다. 기능·버그 수정에 이 우회를 사용하면 안 된다.
- 제품 변경은 관련 집중 테스트, `npm run lint`, `npm test`, `npm run tdd:guard`, `npm run build`를 통과해야 한다. 화면 흐름·라우팅·복구·정답 공개 시점이 바뀌면 `npm run test:e2e`도 통과해야 한다.
- 교육 콘텐츠 변경은 해당 학년 검증기와 `npm run audit:missions` 또는 `npm run audit:problems` 중 영향을 받는 감사를 통과해야 한다. 5학년 JSON 템플릿 변경은 `npm run validate:templates`를 생략할 수 없다.
- 개발 서버와 프로덕션 빌드를 동시에 실행해 같은 `.next` 디렉터리를 공유하면 안 된다. 둘이 겹친 검증 결과는 유효한 통과로 보지 않는다.

## 코드 경계

- `src/app`은 경로와 화면 조립을 소유하며 정답 계산 규칙을 직접 구현하지 않는다. 화면 컴포넌트 안에 수학 판정이 추가되면 위반이다.
- `src/components`는 표현과 입력을 소유하며 localStorage 키를 새로 정의하거나 진도 보상량을 결정하지 않는다.
- `src/lib`는 화면 컴포넌트를 가져오지 않는다. 학습 규칙과 저장 정규화가 화면 순환 의존성을 만들면 위반이다.
- `public/data`의 형식이 바뀌면 `src/lib/types.ts`, 생성기, 검증기, 집중 테스트를 같은 변경에서 맞춘다. 일부 소비자만 새 형식을 이해하면 완료가 아니다.
- `src/lib/types.ts`, `src/lib/problem-generator.ts`, `src/lib/session.ts`, `public/data/concepts.json`, `public/data/templates/*.json`, 배포 작업 파일은 공유 지점이다. 병렬 변경 전 `workstreams/_shared/README.md`에 소유 범위와 계약을 남기지 않으면 동시에 수정하지 않는다.

## 문제와 입력 계약

- 5학년 템플릿은 `set_id` A·B·C와 난이도 1·2·3을 가져야 하며 각 세트는 4·4·2로 검증한다. 객관식은 서로 다른 보기 네 개와 정확히 한 정답을 보장해야 한다.
- 템플릿 표현식은 생성기와 검증기가 모두 평가할 수 있는 등록 함수와 제한된 산술만 사용한다. 평가되지 않은 `[표현식?]`이 생성되면 콘텐츠를 배포할 수 없다.
- 생성과 채점은 결정적이어야 하며 AI 응답을 호출 경로에 넣지 않는다. 자연어 힌트를 추가해도 원래 정답과 규칙 기반 풀이를 덮어쓰면 안 된다.
- 숫자·구조화 입력은 정규화 성공을 확인한 뒤에만 정오를 확정한다. 완성되지 않은 형식을 오답으로 저장하거나 답을 잠그는 테스트가 통과하면 안 된다.
- 그림 데이터의 `result`, `target`, `product`, 정답 치수와 같은 값은 `showAnswer`와 동등한 공개 상태 뒤에서만 렌더링한다. 제출 전 DOM이나 접근성 이름에 답 전용 값이 있으면 위반이다.
- 정량 그림은 문제·정답과 같은 수학 모델에서 만들어야 하며 위상, 비율, 라벨, 0 영역 검증을 추가한다. 고정 SVG 좌표만 눈으로 확인한 정량 그림은 출시할 수 없다.

## 저장 형식

- 기존 localStorage 이름과 읽기 가능한 필드를 변경할 때는 이전 기록을 무손실로 읽는 테스트를 먼저 둔다. 새 형식으로 자동 변환하면서 완료·복습·선택 단원·최근 활동을 잃으면 위반이다.
- 학년 하나의 손상된 기록을 복구할 때 다른 학년 키를 지우지 않는다.
## 협업과 배포

- 작업 전 `workstreams/`에서 주 작업영역을 정하고 그 경계를 우선한다. 공유 계약을 바꾸면 같은 변경에서 `workstreams/_shared/README.md`를 갱신한다.
- 외부 조사 근거는 `references/`, 다른 작업자에게 필요한 상태·위험·다음 행동은 `handoffs/YYYY-MM-DD-workstream-agent.md`에 남긴다.
- 커밋 전 의도한 파일만 명시적으로 스테이징하고 `git diff --cached --check`를 통과시킨다. 관련 없는 기존 변경과 새 파일을 함께 커밋하면 안 된다.
- `main` 반영은 자동으로 실제 GitHub Pages 배포를 시작한다. 검증되지 않은 커밋을 `main`에 푸시하거나 작업 성공만 보고 배포 성공을 가정하면 안 된다.
