# 검증·생성 도구 작업 규칙

## 담당 범위

이 디렉터리는 학년별 문제 은행 검증, 5학년 템플릿 검증과 생성, 문제·미션 품질 감사, 작업 단계 실행을 담당한다. 실제 학습 화면, 런타임 채점 권한, 배포 워크플로 자체는 담당하지 않는다.

## 경계

- 검증기는 제품과 같은 수학·데이터 계약을 검사하되 검증 결과 파일을 런타임 입력으로 만들지 않는다.
- 생성 스크립트는 사람이 검토할 수 있는 결정적 결과를 만들어야 한다. 실행할 때마다 이유 없이 대규모 JSON 순서나 값이 바뀌면 안 된다.
- 품질 보고서는 문제를 설명할 수 있지만 정답 판정을 AI 평가 결과로 대체하지 않는다.
- 제품 타입이나 템플릿 형식이 바뀌면 검증기 오류 코드와 실패 예제를 같은 변경에서 갱신한다.

## 항상 유지할 동작

- 검증 실패는 0이 아닌 종료 코드와 템플릿·미션 식별자, 위반 필드를 제공한다. 오류를 경고로 낮춰 출시를 통과시키지 않는다.
- 모든 허용 매개변수에서 객관식 정답 하나, 보기 중복 없음, 평가 가능한 표현식, 유효 단위·정답을 검사한다.
- 미션 감사는 답 형식, 시각 데이터, 보상 식별자와 장기 시드 다양성을 제품 규칙과 같은 기준으로 본다.
- 정량 시각 검증을 추가할 때 위상 일치, 핵심 대소 관계, ±10% 비율, 라벨 영역 포함, 0 영역 비표시를 각각 구분된 실패로 보고한다.
- 작업 단계 실행기는 `src/**` 또는 `public/data/**` 변경에 집중 테스트가 없으면 정상 완료로 처리하지 않는다.

## 구현 방식

- 공유 검사 로직은 `*-core.js`에 두고 CLI 보고서는 입력·출력과 종료 코드를 조립한다.
- 파일 검색과 출력 경로는 저장소 루트를 기준으로 결정하며 사용자 홈이나 임의의 현재 디렉터리에 결과를 쓰지 않는다.
- 생성기가 JSON을 대량 갱신하면 생성 명령 뒤 검증·감사를 바로 실행하고 diff에서 의도하지 않은 재정렬을 확인한다.

## 테스트

- 새 오류 조건은 통과 예제와 실패 예제를 모두 추가하고 오류 코드·식별자·종료 상태를 검사한다.
- 검증기 자체 변경도 `src/lib/*audit.test.ts` 또는 전용 집중 테스트를 동반한다.
- 전체 확인은 관련 `validate:*`, `audit:*`, `npm test`, `npm run lint`, `npm run build` 순서로 수행한다.
- 외부 품질 평가 도구가 실패해도 결정적 검증 결과와 혼동하지 않는다. 외부 평가 성공만으로 콘텐츠를 통과시키지 않는다.
